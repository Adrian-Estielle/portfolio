Windows Autopilot enrollment + recovery checklist

Operational checklist + recovery patterns for Autopilot enrollment failures and half-enrolled states.
Sanitized note: repeatable runbook format. No tenant IDs, secrets, internal hostnames, customer identifiers.

Why this exists

Autopilot succeeds when enrollment is predictable and recoverable. This runbook focuses on pre-req validation, stability during OOBE, and fast recovery from half-enrolled states.

Enrollment checklist

  1. Assignment sanity: confirm device is targeted to the correct Autopilot profile + required apps/policies.
  2. Network stability: OOBE is sensitive to flaky Wi‑Fi. Prefer Ethernet during initial setup when possible.
  3. MFA readiness: ensure the user can complete MFA on first sign-in (avoid “stuck in limbo” scenarios).
  4. Validate: confirm enrollment completes, required apps install, and policies show as applied.

Recovery (half-enrolled / weird state)

  • Remove stale device objects where appropriate (Intune/Entra) so you don’t keep re-applying policy to a broken record.
  • Wipe and re-enroll cleanly when the state is unrecoverable.
If someone tried to sign in before MFA was ready, you can end up with a partial state that isn’t reliably monitored. Recovery is usually “clean up records → re-enroll clean.”

OOBE troubleshooting

  • Shift+F10 can open a command prompt during OOBE (useful for quick checks like connectivity and logs).
  • Don’t walk around on weak Wi‑Fi during enrollment; lost connectivity can break the flow.
