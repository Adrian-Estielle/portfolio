How I build & validate this portfolio

My proof-first rules: what I publish, how I sanitize it, and what I verify before it goes public.

Proof-first strategy

  • Projects = evidence: code + a reproducible process + an observable output (artifact/report/run log).
  • Field Notes = credibility: I write why → steps → validation → failure modes (including click paths and log locations).
  • Sanitization is non-negotiable: no tenant IDs, internal domains, user identifiers, customer data, or secrets.
Intent: make it easy for a reviewer to verify claims quickly by following links and reproducing outputs.

What counts as proof

  • CI runs with visible logs and downloadable artifacts.
  • Scripts that produce an output file (report, log, exported config) that can be inspected.
  • Runbooks that include “where to look when it fails” (UI clicks, log paths, common error patterns).
  • Before/after evidence when change is the point (migration checklist, validation steps, rollback plan).

Sanitization rules (public-safe)

  • I never publish originals; I publish sanitized copies under docs/.
  • I replace identifiers/secrets with placeholders like <REDACTED_TENANT_ID>, <REDACTED_EMAIL>, <REDACTED_SECRET>.
  • If I can’t confidently sanitize something (binary artifacts, screenshots, unclear context), I mark it NEEDS MANUAL REVIEW and don’t ship it.
Hard stop: if a secret scan hits anything that looks like a live credential, I stop and fix it before pushing.

Validation checklist (what I run before pushing)

  • Link sanity: every new page and evidence link is relative and resolves under docs/.
  • Secret scan: search for common credential patterns (tokens, private keys, internal hostnames).
  • Spot-check: open a handful of artifacts and confirm placeholders are present where expected.
  • Git hygiene: stage only intended files under docs/ and keep scratch/backup files untracked.

How I use GPT via CLI (safely)

  • I use a CLI workflow to accelerate drafting runbooks, generating inventories, and creating repeatable publish scripts.
  • I treat the model output as a draft and verify it against the filesystem and tool output before publishing.
  • I keep the workflow resumable: checkpoints on disk, streaming reads for large inventories, and “resume forward” behavior.
