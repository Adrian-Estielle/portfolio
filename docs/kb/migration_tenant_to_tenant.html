Microsoft 365 tenant-to-tenant migration (high-level runbook)

Inventory → identity mapping → waves → cutover → validation.
Sanitized note: repeatable runbook format. No tenant IDs, secrets, internal hostnames, customer identifiers.

Why tenant-to-tenant needs discipline

Tenant migrations fail when treated like “just move data.” Identity mapping, domain cutover, coexistence, and permissions models are the real work.

High-level plan

  1. Inventory: mailboxes, OneDrive, SharePoint sites, Teams, apps, identities.
  2. Identity mapping: UPNs, groups, shared mailboxes, service principals.
  3. Coexistence strategy: pilot, wave scheduling, user comms.
  4. Move workloads: mail first (or last) depending on constraints; files/sites with permissions mapping.
  5. Cutover domains: minimize downtime; verify auth + mail flow + device sign-in behavior.
  6. Post-cutover validation: access, sharing links, external collaboration settings.

Operational reality

“Success” isn’t “data exists in the new tenant.” Success is “users can work on day 1 without mystery access issues or broken workflows.”
