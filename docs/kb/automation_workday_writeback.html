Workday writeback (runbook notes)

Optional phase 2: write select identity attributes back into Workday. High value when controlled, high risk when sloppy.
* Under active development: this runbook is being expanded with sanitized artifacts, log excerpts, and script snippets.

What writeback is

Writeback pushes selected identity attributes back into Workday. It’s operationally valuable, but it’s a higher-risk change than inbound provisioning because you’re writing into HR data paths.

Stabilize inbound provisioning first. Writeback is phase 2, with tight scope + change control + rollback plan.

When I use it

  • Workday needs to store identity attributes that are standardized downstream (email/UPN).
  • You need closed-loop processes (HR ↔ Identity), with auditability and ownership.

Safe rollout pattern

  1. Implement inbound provisioning (Workday → Entra/AD) and prove it’s stable.
  2. Define exactly which attributes are written back, and who owns them.
  3. Pilot writeback on a tiny population.
  4. Validate in Workday and in Entra provisioning logs.
