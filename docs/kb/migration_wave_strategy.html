Wave strategy + performance validation (avoid migration rework)

Pilot low dependency → validate → migrate high-dependency workloads once performance and behavior are proven.
Sanitized note: repeatable runbook format. No tenant IDs, secrets, internal hostnames, customer identifiers.

Why this matters

The safest migrations start with low-dependency groups to validate behavior and performance before moving heavy-churn departments.

Wave strategy (risk-managed)

  1. Start with teams least dependent on the workload (lowest blast radius).
  2. Use pilots to identify performance gaps, permissions edge cases, client behavior, and workflow breaks.
  3. Tune and validate.
  4. Only then migrate the heavy churn / performance-sensitive teams (ex: Accounting).
Risk note: migrating your hardest department first is how you force rework if platform performance or behavior is off. Pilots prevent “move the hardest segment twice.”

Performance validation (real-world gotcha)

  • Region distance matters (tenant/primary region vs user region).
  • High read/write volumes amplify latency and client sync behavior.
  • Validate with the teams that actually churn data (not just light users).

Archive tiering (prevents everything staying “hot” forever)

  • Hot: SharePoint/OneDrive (daily operations).
  • Warm archive: Azure Files (restricted access, ticket-based retrieval).
  • Cold archive: Blob storage (long-term retention).
