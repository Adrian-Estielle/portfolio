Autopilot / enrollment reliability (Windows + Intune)*

Format: step-by-step runbook + real failure modes and how I remediate them.
  1. Pre-flight identity readiness
    Confirm the user’s identity path is clean before first sign-in (MFA readiness, licensing/assignment, and targeting groups).
    Oddity: If a new hire signs in before MFA is configured, enrollment can land in an inconsistent “half-ready” state.
    My approach: don’t keep poking the zombie object for hours — clean up the device record(s) and re-enroll cleanly after MFA is set.
  2. Hardware + firmware baseline
    Validate TPM/Secure Boot and baseline BIOS/firmware settings that commonly impact encryption/compliance.
  3. Network reliability during OOBE
    Use stable connectivity for the initial provisioning window (wired when possible).
    Oddity: Losing internet during the initial setup window can produce “offline / not fully monitored” behavior.
    If this happens, I treat it as an integrity failure: stabilize network, then reset/re-run enrollment instead of guessing.
  4. Provisioning flow + validation
    Confirm expected profiles/apps land and that the device reaches a known-good baseline (compliance, encryption escrow, update ring, Defender state).
    Field note: When local troubleshooting is required during OOBE, I use the standard Windows setup escape hatch (commonly Shift+F10) to validate network state and gather evidence — then I return to the supported path (not permanent hacks).

Compliance triage (“Not compliant” phone/laptop)*

Goal: resolve common compliance failures fast and consistently, without guesswork.
  1. Confirm it’s current — check last sync / last check-in; force a sync if it’s stale.
  2. Identify the failing control — encryption, OS version, Defender, firewall, required apps, jailbreak/root, etc.
  3. Choose the right lever — user-fixable, IT remediation, re-enroll, or wipe (fast + correct beats heroic guessing).
  4. Re-evaluate — confirm policy re-check and that compliance flips with evidence (timestamped).
  5. Escalation threshold
    If it’s clearly a corrupted state (token/auth corruption, broken enrollment, repeated failure after known fixes), stop looping and re-enroll cleanly.
    Common culprit examples:
    • Clock/time drift → token/cert failures → compliance drift.
    • BitLocker/encryption not enabled/escrowed → fix baseline prerequisites first.
    • OS update backlog → remediate update health (storage, servicing stack, reboot loops) then re-check.

SharePoint library auto-sync via Intune (reduce “sync my files” tickets)*

Pattern used to automatically sync specific SharePoint libraries into File Explorer for targeted users (e.g., Accounting).
  1. Define the target outcome — which team/library should appear automatically in File Explorer, and who should receive it.
  2. Gather required identifiers (sanitized)
    Collect the SharePoint site/library identifiers needed for automation (via scripting/APIs).
    Implementation detail (sanitized): the automation is driven by IDs + group targeting, not by manual “click Sync” steps. That’s what removes the support bottleneck.
  3. Model targeting — map library → security group(s) so membership controls who gets the sync by default.
  4. Deploy via Intune — apply the OneDrive/Sync configuration via policy/script, scoped to the intended users/groups.
  5. Validate + remediate OneDrive edge cases
    Pilot with a small set first, verify it survives reboots and user sign-in/out.
    Oddity: If OneDrive is “stuck” (cached auth, broken state), the fix is usually a controlled reset + policy re-apply, not random toggling.

OEM debloat baseline (stability + trust)*

Practical endpoint hygiene: remove known-problem vendor utilities early; verify Office/performance/driver stability.
  1. Maintain a vendor removal list — small and intentional (avoid ripping out things blindly).
  2. Remove early in provisioning — before users start working and before “mystery issues” become normalized.
  3. Validate critical workflows — Office/Excel behavior, performance, drivers, and security posture.
  4. Document the “why”
    Portfolio-grade notes focus on outcomes: stability, reduced support load, and consistent baseline behavior.
    Examples I’ve encountered: HP security suites causing friction, Lenovo utility overhead, etc. (Kept generic in public docs; specifics can be discussed in interviews.)

HelloBuild CI validation (GitHub Actions + Jenkins local pipeline)*

Purpose: “proof-first” CI. GitHub Actions artifacts + Jenkins run (controller in Docker + Windows agent execution).
  1. GitHub Actions: run + artifacts
    Trigger a run (push) and validate a green workflow with artifacts attached.
  2. Jenkins controller: run locally
    Run Jenkins in Docker (ports 8080/50000) and complete initial setup.
    Oddity: If the job defaults to master, explicitly set the branch to main or the pipeline will pull the wrong branch.
  3. Force execution on the Windows agent
    Set built-in executor count to 0 and ensure the job runs on the node labeled win-agent.
    Expected proof string: build logs should say “Running on win-agent”.
  4. Prereqs on the Windows agent
    Confirm the agent environment has what the build needs (Git + .NET SDK + PowerShell).
    Oddity: dotnet not found ⇒ install .NET SDK on the agent (or fix PATH for the agent session).
    This is the difference between a “controller runs” demo and a real “Windows execution” proof.
  5. Run HelloBuild
    Build Now → confirm stages complete → confirm artifacts archived.
  6. Capture proof
    Save a green Stage View / SUCCESS screenshot and keep it in the portfolio as CI proof (already done via docs/images/jenkins_pipeline.png).

Onboarding automation (matrix-driven groups + standardized provisioning)*

Evolution: manual copy process → standardized group matrix → GUI-driven lifecycle tool.
  1. Collect essentials: first/last, title, office/region, manager.
  2. Generate identity: username/email based on convention; set manager; set domain defaults.
  3. Apply access: use a title/region matrix to assign default security groups consistently.
  4. Stop relying on “copy user” forever
    Initial phase can copy a peer; mature phase uses a curated default matrix to avoid inherited junk.
    Why this matters: copying accounts preserves legacy mistakes; matrices are auditable and repeatable.
  5. Export evidence: output the applied groups/roles and store logs for auditability.

Termination workflow automation (fast lockout + controlled evidence retention)*

Focus: speed, correctness, and a clean separation between “immediate lockout” and “destructive cleanup.”
  1. Immediate lockout: disable sign-in, revoke sessions, block access.
  2. Preserve business continuity: mailbox handling (hold/retention as required), auto-reply to manager, delegate mailbox access.
  3. Strip entitlements: remove group memberships/privileges; move to a Terminated OU.
  4. Timed deletion + approval checkpoint
    Queue the account for deletion after a defined window; require admin approval before irreversible actions.
    Oddity: MFA method cleanup and destructive removal should be gated — it’s easy to break retention/legal requirements if rushed.

Pre-seed + delta cutover strategy (reduce cutover risk)*

Goal: move most data in advance, then only migrate deltas during cutover.
  1. Discovery: scope, permissions model, file counts/sizes, and risk items (large PSTs, legacy perms, broken inheritance).
  2. Pilot: migrate a representative subset and validate outcomes end-to-end.
  3. Pre-seed early: copy bulk data ahead of cutover using appropriate tooling (kept generic here).
  4. Delta often: run periodic incremental updates so cutover day is small.
  5. Freeze + final delta: enforce a change window, then apply final incremental updates.
  6. Validate: spot checks + counts + integrity checks; document results.

Spreadsheet hyperlink repair / remover (migration link hygiene)*

Use-case: too many old links to realistically redirect during a migration window; update links safely at scale.
  1. Work on a copy: use dev/test copies of the data, never live production as your first testbed.
  2. Parameterize old/new locations: supply old path + new path so only intended links are rewritten.
  3. Handle hidden sheets safely
    Unhide → repair links → re-hide so output stays behaviorally consistent.
    Quality constraint: do not corrupt files and do not break formulas that shouldn’t change.
  4. Preserve timestamps when needed: retain last-modified dates for audit/history expectations.
  5. Validate at scale: sample verification + rollback plan if output deviates.

Cheat sheet*

Fast reference notes that prevent common mistakes.

Permissions: RBAC vs sharing vs NTFS/SMB (mental model)*

If you don’t know which control plane you’re troubleshooting, you will waste time.
  • RBAC controls administrative capability (who can manage).
  • Sharing controls resource access (who can access what).
  • NTFS/SMB effective access can be destroyed by deep inheritance breaks + direct user perms.
  • Rule of thumb: group-based access scales; direct user perms create offboarding debt.
